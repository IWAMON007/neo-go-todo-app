### タスク: 1-1 API通信の共通化（apiFetch ヘルパー関数の作成）

#### ジェネリクス `<T>` の基本

- `<T>` は「型の引数」。関数の引数が値を受け取るように、**型自体を外から受け取る**仕組み
- 定義側は `<T>` をプレースホルダーとして宣言するだけで、中身は呼び出し側が決める
- 呼び出し側が `apiFetch<Todo[]>(...)` のように型を渡すことで `T = Todo[]` が確定する
- `T` は慣習的な名前（Type の略）。`Data` や `Response` でも動くが、`T` が一般的

```typescript
// 定義側: T が何かは知らない
async function apiFetch<T>(params: Params): Promise<T> { ... }

// 呼び出し側: T を指定して型を確定させる
const list = await apiFetch<Todo[]>({ ... })
```

#### `async / await / Promise` の流れ

- `async` 関数を定義すると、戻り値は自動的に `Promise` でラップされる
- `await` は `Promise` が解決されるまでその行で待ち、中身の値を取り出す
- `await` を使うには、その関数自体も `async` である必要がある
- `Promise<void>` は「値を返さず、副作用だけ行う非同期関数」の型

```typescript
// Promise<void>: 戻り値なし、todoList の更新だけを副作用として行う
export async function getTodoList(): Promise<void> {
    const list = await apiFetch<Todo[]>({ ... })
    if (list) todoList.value = list
}
```

#### 無名関数の使いどころ

- `async () => {}` は関数を定義するだけで実行しない（よくあるミス）
- 即時実行したい場合は `(async () => { ... })()` と末尾に `()` をつける（IIFE）
- 無名関数が本当に輝くのは「引数として渡すとき」（`setTimeout`, `onMounted`, `Array.map` など）
- 単純な `async` 処理には `async function` を使う方が読みやすい

#### モジュールスコープの状態共有

- `ref` を関数の外（モジュールスコープ）に置くと、全コンポーネントで同じインスタンスを共有できる
- 関数の中に置くと呼ぶたびに新しい `ref` が生まれるため、状態が共有されない
- `export const todoList = ref<Todo[]>([])` のように直接 `export` することで、コンポーネントから直接 import できる

```typescript
// 関数の外 → 全コンポーネントで共有される同一インスタンス
export const todoList = ref<Todo[]>([])

// 関数の中 → 呼ぶたびに新しい ref が生まれる（共有されない）
export function useXxx() {
    const todoList = ref<Todo[]>([])  // ← これは共有されない
}
```

#### 旧設計 vs 新設計の比較

| | 旧（useGetTodoList） | 新（getTodoList + export const） |
|---|---|---|
| `todoList` の取り出し方 | `useGetTodoList()` の戻り値から分割代入 | 直接 `export` して import |
| `getTodoList` | composable 関数の内部に定義・返す | 単独の `async` 関数として `export` |
| コンポーネント側 | `const { todoList, getTodoList } = useGetTodoList()` | `import { todoList, getTodoList } from '...'` |

#### Claude からの FB

- ジェネリクスの「定義側は型を知らず、呼び出し側が決める」という分業の感覚を自力で掴めた
- `async/await/Promise` の流れを正確に言語化できており、理解度は高い
- 無名関数を使おうとした発想は良い（利便性の実感）が、「定義と実行の違い」でハマった。典型的なミスなので今後は意識する
- モジュールスコープの設計意図（なぜ関数の外に置くのか）を自分で問えている点が良い

### タスク 1-4b: Doneボタン押下時のバグ修正

**バグの内容**
- DoneButton をクリックしてもタスクが Home から消えない
- クリックするたびに完了済みTodoが追加される（重複追加）

**原因**
`TaskTable.vue` / `DoneTaskTable.vue` で `v-for` ループ内に `v-if` による `IsDone` フィルタリングがなかった。
前回の `refactor/task-table-split` でテーブルを分割した際に、フィルタリング処理が抜け落ちていた。

**修正内容**
1. 最初の修正: `<tr v-if="!todo.IsDone">` / `<tr v-if="todo.IsDone">` を追加
2. レビュー後の改善: `computed` でフィルタ済みリストを作成し、`v-for` 対象を絞り込む形に変更

```ts
// TaskTable.vue
const activeTodoList = computed(() => todoList.filter((todo) => !todo.IsDone));

// DoneTaskTable.vue
const doneTodoList = computed(() => todoList.filter((todo) => todo.IsDone));
```

**ポイント**
- `v-for` + `v-if` を同じ要素に書くと全件ループ後に表示を切り替えるだけで非効率
- `computed` でフィルタリングすることで、ループ対象を最初から絞れる

## 自分のメモ

### Done.vue の async/await を削除した判断

`onMounted(async () => { await getTodoList(); })` の `async/await` を削除。

**理由**: `apiFetch` 内で `try/catch` が完結しているため、呼び出し側で `await` してエラーを受け取る必要がない。`todoList` は `ref` でリアクティブに共有されているため、`getTodoList()` が完了次第 UI に反映される。

**ポイント**: `await` が必要なのは「呼び出し側でエラーをハンドリングしたい」「完了を待って次の処理をしたい」場合。どちらも不要なら省略して問題ない。

### タスク 1-5: 型安全性の改善（`useFetch.ts`）

**修正内容**
- `if (!error)` の dead code を削除（`catch` ブロックでは常に何かしら値が入るため常に `false`）
- `instanceof Error` による型ガードを実装
- 非 `Error` な例外（文字列など）は `new Error('予期せぬエラー')` でラップして投げる
- `!response.ok` のエラーメッセージを `params.error.message`（呼び出し元固有）に変更

**`any` vs `unknown` の整理**
- `any`: プロパティに型チェックなしでアクセスできてしまう（危険）
- `unknown`: 型ガードするまでプロパティにアクセスできない（安全）
- `throw` は TypeScript の型チェック対象外のため `unknown` のまま `throw` できる
- `!error` のような真偽値評価は `unknown` でも許可されている（プロパティアクセスのみ禁止）

**設計メモ**
- `fetch()` はサーバーが 4xx/5xx を返しても例外を投げない → `!response.ok` で意図的に例外に変換する必要がある
- より厳密にするには HTTP ステータスごとにカスタム `Error` 型を作る設計が有効（タスク 1-7 で検討予定）

### テーマ: 非同期処理におけるエラーの伝播と設計

#### エラーは throw すると呼び出し元に伝わる

`apiFetch` 内で `throw` すると、`await` している呼び出し元にエラーが伝播する。
呼び出し元が `try/catch` していなければ、さらにその上へ伝わり続ける。

```typescript
// apiFetch が throw すると…
export async function addTask() {
    const addedTask = await apiFetch<Todo>(...)  // ← ここで伝播してくる

    newTask.value = ''           // ← 実行されない
    todoList.value.push(...)     // ← 実行されない
}
```

#### エラーの伝播経路

```
apiFetch → throw
  ↓
addTask → catch なし → 素通り
  ↓
FormTask.vue → catch なし → 素通り
  ↓
Home.vue → catch なし → 素通り
  ↓
Vue のグローバルエラーハンドラーで止まる
```

#### 「通知」と「処理を止める」は別の関心事

- 今の設計は `apiFetch` の中で `alert` を出しているので、ユーザーへの通知はできている
- ただしそれは「たまたまそこで通知している」だけ
- エラーをどこで**通知**して、どこで**処理を止める**かを意識して設計することが重要

| 場所 | 役割の例 |
|---|---|
| `apiFetch` | 通信エラーの検知・ログ出力 |
| `addTask` など | ビジネスロジックの失敗処理（フォームのリセットをしない、など） |
| コンポーネント | UI の状態をエラー表示に切り替える |

#### 今の設計のトレードオフ

- 「`apiFetch` が全部やる」に寄っているのでシンプル
- 柔軟性は低い（コンポーネントごとに異なるエラーUIを出したい場合などに対応しにくい）
- 学習段階ではこれで十分。実務では「どこで止めるかを設計する」視点が必要になる

### テーマ: onErrorCaptured の仕組みと使いどころ

#### onErrorCaptured の基本

- Vue の `onErrorCaptured` は**子コンポーネントから伝播してきたエラー**をキャッチするライフサイクルフック
- `return false` を返さないと、エラーが Vue のグローバルエラーハンドラーまで伝播し続ける
- `<script setup>` の中で呼ばないと "no active component instance" の警告が出て機能しない

#### コンポーネント構造とエラーの受け取り先

```
App.vue          ← getTodoList（Home.vue自身のonMounted）のエラーはここで拾う
  └── Home.vue   ← onErrorCaptured を置く
        ├── FormTask.vue   ← addTask のエラー → Home.vue が拾う ✓
        └── TaskTable.vue  ← doneTask 等のエラー → Home.vue が拾う ✓
```

- **同じコンポーネント自身のエラーは自分の `onErrorCaptured` では拾えない**（親が必要）

#### onMounted で Promise を Vue に渡す書き方

```typescript
// NG: Promise を返していない → Vue がエラーを追跡できない → unhandled rejection になる
onMounted(() => {
    getTodoList()
})

// OK: Promise を返す → Vue が callWithAsyncErrorHandling でラップする
onMounted(() => getTodoList())
```

#### async/await と Vue のライフサイクルフック

- `onMounted(async () => { await fn() })` は技術的に動くが、副作用の懸念から避けるスタイルもある
- `onMounted(() => fn())` のように Promise を返す形が安全でシンプル

### テーマ: エラーの種類の区別と TypeScript の unknown

#### try/catch 内の throw の挙動

- `try` ブロック内で `throw` した瞬間、残りの処理は**実行されず即座に `catch` に飛ぶ**
- `catch` に渡される `error` は `throw` したオブジェクトそのもの

```typescript
try {
    throw new Error('サーバーエラー')  // ← ここで throw
    // 以降は実行されない
} catch (error) {
    // ← 即座にここへ
}
```

#### unknown 型と instanceof による型の絞り込み

- `catch (error: unknown)` はコンパイル時の型。実行時には実際の値が入っている
- `instanceof` は**実行時**に型を確認するチェック
- `unknown` に対して `instanceof TypeError` で型を絞り込むのが TypeScript の正しいアプローチ

```typescript
catch (error: unknown) {
    if (error instanceof TypeError) {
        // fetch 自体が失敗（ネットワーク断）→ TypeError: Failed to fetch
        throw new Error('通信エラーが発生しました。接続を確認してください。')
    }
    // !response.ok で throw した Error → そのまま再 throw
    // → 「サーバーエラーが発生しました」が伝わる
    throw error
}
```

#### ネットワークエラー vs サーバーエラーの区別

| 状況 | 原因 | エラーの型 |
|---|---|---|
| `fetch` 自体が失敗 | ネットワーク断・サーバー停止 | `TypeError: Failed to fetch` |
| `response.ok` が false | サーバーが 4xx/5xx を返した | `Error`（自分で throw したもの） |

### テーマ: フレームワークなし構成でのエラーハンドリング設計

#### フレームワークが「よしなに」してくれること

- Next.js / Nuxt.js などは 404/500 専用ページへの遷移・認証エラーのリダイレクトなどを自動でやってくれる
- ただし「このステータスのときはこう動く」という**設定は結局自分で書く**

#### 今の構成（vanilla Vue + Go）での方針

- HTTP ステータスコードごとに `apiFetch` で場合分けするのが愚直だが正しいアプローチ
- フレームワークも内部では同じことをやっている。自分で書くことで仕組みの理解が深まる

#### 現状の粒度

今は「ネットワーク系 / サーバー系」の2分類で十分。より細かくしたい場合：

```typescript
if (response.status === 401) throw new Error('認証エラー')
if (response.status === 404) throw new Error('リソースが見つかりません')
if (response.status >= 500) throw new Error('サーバーエラー')
```
